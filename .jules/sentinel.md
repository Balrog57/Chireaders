## 2025-06-03 - [Race Condition in AsyncStorage Background Tasks]
**Vulnerability:** A "Check-Then-Act" race condition was found in `BackgroundNotificationTask.js`. The task read data from `AsyncStorage`, performed long-running network operations, and then overwrote the storage with the stale initial data plus updates. This caused user changes (e.g., deleting a favorite) made during the task execution to be lost.
**Learning:** Background tasks in React Native headless JS operate independently of the UI state and contexts. Direct `AsyncStorage` usage without re-validation is dangerous for data integrity when user interaction is possible concurrently.
**Prevention:** Always implement a "Fetch-Merge-Save" strategy for background tasks: collect updates first, then re-read the storage source of truth immediately before saving, merging the updates into the fresh state.

## 2025-06-03 - [WebView Message Origin Vulnerability]
**Vulnerability:** A missing origin validation was found in `BrowserScreen.js` within the `handleMessage` handler for `react-native-webview`. Messages posted from the WebView (via `window.ReactNativeWebView.postMessage`) were parsed and processed without checking the URL/origin they came from.
**Learning:** React Native WebViews allow bidirectional communication with JavaScript. If a user navigates to an attacker-controlled page, or if a legitimate site is compromised, an attacker can execute malicious JS and use `postMessage` to send forged events. Without origin checks, the native app might trust these events, potentially leading to unauthorized actions or data leakage depending on what the message handlers do.
**Prevention:** Always use strict domain validation on `event.nativeEvent.url` within the `onMessage` handler of a WebView before parsing or trusting the message contents. Only process messages if the URL exactly matches an expected, trusted origin.
