## 2025-06-03 - [Race Condition in AsyncStorage Background Tasks]
**Vulnerability:** A "Check-Then-Act" race condition was found in `BackgroundNotificationTask.js`. The task read data from `AsyncStorage`, performed long-running network operations, and then overwrote the storage with the stale initial data plus updates. This caused user changes (e.g., deleting a favorite) made during the task execution to be lost.
**Learning:** Background tasks in React Native headless JS operate independently of the UI state and contexts. Direct `AsyncStorage` usage without re-validation is dangerous for data integrity when user interaction is possible concurrently.
**Prevention:** Always implement a "Fetch-Merge-Save" strategy for background tasks: collect updates first, then re-read the storage source of truth immediately before saving, merging the updates into the fresh state.

## 2025-06-03 - [WebView Message Origin Validation]
**Vulnerability:** The React Native `WebView` component receives messages from loaded web pages via `onMessage`. The `handleMessage` handler in `BrowserScreen.js` processed all incoming messages without verifying the origin URL (`event.nativeEvent.url`). This allows any malicious site loaded in the WebView (or a compromised legitimate site) to send unauthorized instructions (like `styleApplied` or `titleChanged`) that the host application blindly executes.
**Learning:** Never trust data received from a `WebView` implicitly, even if the expected message types seem benign. The origin of every incoming message must be verified to prevent injection or unexpected behavior from untrusted domains.
**Prevention:** Always implement an origin check (e.g., using `isValidChiReadsUrl(event.nativeEvent.url)`) as the very first step in the `WebView`'s `onMessage` handler. Discard and optionally log any messages originating from unauthorized URLs.
